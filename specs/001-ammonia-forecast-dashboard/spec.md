
# 機能仕様書: アンモニア在庫レベル予測ダッシュボード

**Feature Branch**: `001-ammonia-forecast-dashboard`  
**Created**: 2025-11-03  
**Status**: Draft  
**Input**: User description: "アンモニア在庫レベル予測ダッシュボード - インタラクティブなチャート、日付範囲選択、補充警告システム、統計情報表示を備えたNext.js製ダッシュボード"

## User Scenarios & Testing *(必須)*

### User Story 1 - 在庫予測データの可視化 (Priority: P1)

運用担当者は、アンモニア在庫の実績値と予測値を時系列グラフで確認し、在庫レベルの推移を把握する。

**Why this priority**: 在庫状況の可視化は、補充計画と異常検知の基盤となる最も重要な機能。この機能なしでは、ダッシュボードの価値が存在しない。

**Independent Test**: 予測データCSVファイルをロードし、基準日を中心とした60日間（前後30日ずつ）の在庫推移グラフが表示されることを確認する。実績値と予測値の2本の線が区別可能な形で描画され、凡例が表示される。

**Acceptance Scenarios**:

1. **Given** 予測データCSVファイルが`/public/data/predictions.csv`に配置されている状態で、**When** ダッシュボードを開く、**Then** 基準日（デフォルト：本日または最も近い日付）を中心とした60日間のグラフが表示される
2. **Given** グラフが表示されている状態で、**When** グラフ上の任意のポイントにマウスをホバーする、**Then** その日付の実績値、予測値、誤差、誤差率がツールチップで表示される
3. **Given** グラフが表示されている状態で、**When** 補充が実施された日付を確認する、**Then** 補充日にマーカーまたは視覚的な区別が表示される
4. **Given** データ範囲が2024年10月1日から2025年10月31日の期間である状態で、**When** 基準日を2024年12月15日に設定する、**Then** 2024年11月15日から2025年1月14日のデータがグラフに表示される

---

### User Story 2 - 日付範囲の選択と調整 (Priority: P1)

運用担当者は、基準日を変更することで、過去の実績や未来の予測を柔軟に表示し、任意の期間の在庫推移を分析する。

**Why this priority**: 時系列データ分析において、表示期間の調整は必須機能。過去の補充タイミングの評価や未来の補充計画に直接影響する。

**Independent Test**: 基準日入力フィールドに任意の日付を入力し、その日付を中心とした60日間のグラフが再描画されることを確認する。データ範囲外の日付を入力した場合、最も近い有効な日付に自動クランプされる。

**Acceptance Scenarios**:

1. **Given** ダッシュボードが表示されている状態で、**When** 基準日入力フィールドに「2024-11-15」と入力する、**Then** グラフが2024年10月16日から2024年12月15日の期間に更新される
2. **Given** ダッシュボードが表示されている状態で、**When** 基準日にデータ範囲外の日付（例：2023年1月1日）を入力する、**Then** データの最初の日付（2024年10月1日）に自動クランプされ、警告メッセージが表示される
3. **Given** ダッシュボードが表示されている状態で、**When** 基準日にデータ範囲外の未来の日付（例：2026年1月1日）を入力する、**Then** データの最終日付（2025年10月31日）に自動クランプされ、警告メッセージが表示される
4. **Given** グラフが表示されている状態で、**When** ブラウザをリロードする、**Then** 基準日は本日または最も近い有効な日付にリセットされる

---

### User Story 3 - 補充警告システムの自動検知 (Priority: P1)

運用担当者は、在庫レベルが設定した閾値を下回る予測が検出された際に、自動的に警告通知を受け取り、補充タイミングを把握する。

**Why this priority**: 在庫切れを防ぐための予防的アクションの起点となる機能。補充の遅れはプラントの稼働停止に直結する可能性がある。

**Independent Test**: 補充閾値を600 kgに設定し、基準日を中心とした表示期間内で予測在庫が600 kgを下回る日付が存在する場合、警告バナーが表示され、初めて閾値を下回る日付が強調表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 補充閾値が600 kgに設定されており、表示期間内に予測在庫が600 kgを下回る日が存在する状態で、**When** ダッシュボードを開く、**Then** 画面上部に「⚠️ 補充警告」バナーが表示される
2. **Given** 補充警告が表示されている状態で、**When** 警告バナーを確認する、**Then** 初めて閾値を下回る日付（例：2024年12月8日）と、その日の予測在庫量が表示される
3. **Given** 補充閾値入力フィールドが表示されている状態で、**When** 閾値を500 kgに変更する、**Then** グラフ上の閾値ラインが500 kgに更新され、警告判定が再計算される
4. **Given** 表示期間内にすべての予測在庫が閾値以上である状態で、**When** ダッシュボードを確認する、**Then** 補充警告バナーは表示されない

---

### User Story 4 - 統計情報の表示 (Priority: P2)

運用担当者は、予測モデルの精度指標（R²、平均誤差）、現在の在庫レベル、次回補充推奨日を一目で確認し、意思決定の参考にする。

**Why this priority**: 予測の信頼性を評価し、補充計画の確実性を高めるために重要。ただし、グラフと警告システムがあれば最低限の運用は可能。

**Independent Test**: ダッシュボード上部または側面に統計情報カードが表示され、以下の情報が含まれることを確認する：現在の在庫レベル、予測精度（R²）、平均誤差、次回補充推奨日。

**Acceptance Scenarios**:

1. **Given** ダッシュボードが表示されている状態で、**When** 統計情報カードを確認する、**Then** 基準日時点の在庫レベル（実績値または予測値）が「現在の在庫」として表示される
2. **Given** 統計情報カードが表示されている状態で、**When** 予測精度を確認する、**Then** R²値がパーセンテージ形式（例：85%）で表示される
3. **Given** 統計情報カードが表示されている状態で、**When** 平均誤差を確認する、**Then** 予測誤差の平均値が絶対値またはパーセンテージで表示される
4. **Given** 補充警告が発生している状態で、**When** 統計情報カードを確認する、**Then** 次回補充推奨日が表示される（例：2024年12月8日）
5. **Given** 基準日を変更した状態で、**When** 統計情報カードを確認する、**Then** すべての統計値が新しい基準日に基づいて再計算される

---

### User Story 5 - レスポンシブUIとダークモード (Priority: P3)

運用担当者は、モバイル、タブレット、デスクトップのいずれのデバイスからでもダッシュボードにアクセスし、視覚的に快適なダークモードで在庫データを確認する。

**Why this priority**: ユーザー体験の向上と、多様なデバイスでのアクセシビリティを確保する。ただし、機能的な価値はP1-P2に比べて低い。

**Independent Test**: スマートフォン、タブレット、デスクトップの各画面サイズでダッシュボードを表示し、レイアウトが適切に調整されることを確認する。ダークモードが適用され、ネオングラデーションUIが表示される。

**Acceptance Scenarios**:

1. **Given** ダッシュボードをスマートフォン（幅375px）で開いた状態で、**When** 画面を確認する、**Then** グラフと統計カードが縦方向に配置され、スクロール可能である
2. **Given** ダッシュボードをタブレット（幅768px）で開いた状態で、**When** 画面を確認する、**Then** グラフと統計カードが適切な2カラムレイアウトで表示される
3. **Given** ダッシュボードをデスクトップ（幅1920px）で開いた状態で、**When** 画面を確認する、**Then** グラフが画面幅を最大限活用し、統計カードがサイドバーに配置される
4. **Given** ダッシュボードが表示されている状態で、**When** ページのスタイルを確認する、**Then** ダークモード（背景が暗色）とネオングラデーションが適用されている

---

### Edge Cases

- **データファイルが存在しない、または空の場合**: エラーメッセージを表示し、「データファイルが見つかりません。`/public/data/predictions.csv`を確認してください」と通知する
- **CSVファイルのフォーマットが不正な場合**: パースエラーを検知し、「CSVファイルの形式が正しくありません。必要な列（date, actual_ammonia, predicted_ammonia）を確認してください」と通知する
- **基準日がデータ範囲外の場合**: 最も近い有効な日付にクランプし、警告メッセージを表示する（US2-AS2, AS3でカバー）
- **補充閾値に無効な値（負の数、文字列など）が入力された場合**: 入力をブロックし、「有効な数値（0以上）を入力してください」とフィードバックを表示する
- **ブラウザがChart.jsをサポートしていない場合**: 互換性エラーを表示し、「お使いのブラウザは推奨されていません。Chrome、Firefox、Edge、またはSafariをご利用ください」と通知する
- **データに欠損値（null、空文字）が含まれる場合**: 欠損値を持つ行をスキップし、コンソールに警告ログを出力する（グラフの連続性は保つ）

## Requirements *(必須)*

### Functional Requirements

- **FR-001**: System MUST CSVファイル（`/public/data/predictions.csv`）から在庫予測データを読み込み、パースする
- **FR-002**: System MUST 基準日を中心とした前後30日間（合計60日間）のデータをフィルタリングする
- **FR-003**: System MUST 実績在庫値（actual_ammonia）と予測在庫値（predicted_ammonia）を2本の線グラフとして描画する
- **FR-004**: System MUST 補充が実施された日付（is_refill = 1）を視覚的に区別する（マーカーまたは色の変更）
- **FR-005**: System MUST グラフ上のポイントにマウスホバー時、日付、実績値、予測値、誤差、誤差率をツールチップで表示する
- **FR-006**: System MUST 基準日入力フィールドを提供し、ユーザーが任意の日付を入力可能にする
- **FR-007**: System MUST 基準日が変更された際、グラフと統計情報をリアルタイムで更新する
- **FR-008**: System MUST 基準日がデータ範囲外の場合、最も近い有効な日付に自動クランプし、警告メッセージを表示する
- **FR-009**: System MUST 補充閾値入力フィールドを提供し、デフォルト値を600 kgとする
- **FR-010**: System MUST 表示期間内の予測在庫が補充閾値を下回る場合、警告バナーを表示する
- **FR-011**: System MUST 警告バナーに、初めて閾値を下回る日付と予測在庫量を含める
- **FR-012**: System MUST グラフ上に補充閾値を示す水平ラインを描画する
- **FR-013**: System MUST 基準日時点の在庫レベルを統計カードに表示する
- **FR-014**: System MUST 予測モデルの精度指標（R²値）を統計カードに表示する
- **FR-015**: System MUST 予測誤差の平均値を統計カードに表示する
- **FR-016**: System MUST 次回補充推奨日（閾値を下回る最初の日）を統計カードに表示する
- **FR-017**: System MUST レスポンシブデザインを実装し、スマートフォン（375px〜）、タブレット（768px〜）、デスクトップ（1024px〜）で適切に表示する
- **FR-018**: System MUST ダークモードのUIを適用し、ネオングラデーションを使用する
- **FR-019**: System MUST CSVファイルの読み込みエラー、パースエラー、データ欠損に対して適切なエラーハンドリングを実装する
- **FR-020**: System MUST 日本時間（JST、UTC+9）でデータを処理し、表示する

### Security Requirements *(憲法準拠: セキュリティ優先の設計)*

**N/A - 本機能は機密データを扱わない**

本ダッシュボードは、アンモニア在庫予測データの可視化のみを目的としており、以下の理由からセキュリティ要件の適用対象外とする：

- ユーザー認証・認可機能は含まれない（公開ダッシュボード）
- 個人情報、認証情報、機密データの保存・送信は行わない
- データは静的CSVファイルから読み込まれ、外部APIとの通信は発生しない
- ユーザー入力は基準日と補充閾値のみであり、サーバーサイド処理は存在しない

**想定される将来的なセキュリティ要件**:

認証機能やAPIアクセスを追加する場合、以下の要件を適用する：

- **SEC-001**: System MUST ユーザー認証にOAuth2またはセッションベース認証を実装する
- **SEC-002**: System MUST すべてのAPI通信をHTTPS/TLSで暗号化する
- **SEC-003**: System MUST CSRFトークンを実装し、クロスサイトリクエストフォージェリを防止する

### Performance Requirements *(憲法準拠: パフォーマンスの定量化)*

- **PERF-001**: System MUST CSVファイル（最大500行、約50KB）を3秒以内に読み込み、パースする
- **PERF-002**: System MUST 基準日変更時、グラフと統計情報を1秒以内に更新する（ユーザー体感的に即座）
- **PERF-003**: System MUST 初回ページロード時、ファーストコンテントフルペイント（FCP）を2秒以内に達成する
- **PERF-004**: System MUST 10,000データポイント（約27年分の日次データ）まで、グラフ描画とインタラクションが遅延なく動作する
- **PERF-005**: System MUST 同時100ユーザーのアクセスに対して、平均応答時間500ms以内でページを配信する（静的ホスティング想定）
- **PERF-006**: System MUST ブラウザのメモリ消費量を通常動作時200MB以下に抑える

**検証方法**:
- PERF-001, PERF-002: ブラウザの開発者ツール（Network、Performance タブ）で計測
- PERF-003: Lighthouse のパフォーマンススコアで計測（目標スコア 90以上）
- PERF-004: 10,000行のテストCSVを使用した手動テスト
- PERF-005: k6、Apache Bench、または Lighthouse CI で負荷テスト
- PERF-006: Chrome DevTools の Memory Profiler で計測

### Key Entities

- **予測データレコード**: 特定の日付における在庫予測情報を表す。以下の属性を含む：
  - 日付（date）: データポイントの日時（JST）
  - 実績電力量（actual_power）: 実際に消費された電力量（kWh）
  - 実績在庫（actual_ammonia）: 実際のアンモニア在庫量（kg）
  - 補充フラグ（is_refill）: 補充が実施されたかどうか（0/1）
  - 予測在庫（predicted_ammonia）: 予測モデルによる在庫量（kg）
  - 予測誤差（prediction_error）: 予測値と実績値の差（kg）
  - 予測誤差率（prediction_error_pct）: 予測誤差の割合（%）

- **基準日**: ユーザーが設定する中心日付。この日付を基準に前後30日間のデータがグラフに表示される。デフォルトは本日または最も近い有効な日付。

- **補充閾値**: 在庫が不足していると判断する基準値（kg）。予測在庫がこの値を下回ると警告が発生する。デフォルトは600 kg。

- **統計情報**: ダッシュボードに表示される集約データ。以下を含む：
  - 現在の在庫レベル: 基準日時点の在庫量
  - 予測精度（R²）: モデルの決定係数
  - 平均誤差: 予測誤差の平均値
  - 次回補充推奨日: 閾値を下回る最初の日付

## Success Criteria *(必須)*

### Measurable Outcomes

- **SC-001**: 運用担当者は、ダッシュボードを開いてから5秒以内に、現在の在庫状況と今後60日間の予測を把握できる
- **SC-002**: 運用担当者は、任意の日付を基準日に設定し、3クリック以内で過去または未来の在庫推移を確認できる
- **SC-003**: 在庫が補充閾値を下回る予測がある場合、運用担当者は警告通知により10秒以内に補充が必要な日付を特定できる
- **SC-004**: ダッシュボードは、スマートフォン、タブレット、デスクトップのいずれのデバイスからでも、レイアウト崩れなく表示される（100%のデバイスサポート）
- **SC-005**: ユーザーの90%以上が、グラフのインタラクション（ホバー、ズーム、日付変更）を初見で理解し、操作できる
- **SC-006**: CSVファイルに500行のデータが含まれる場合でも、ページロード時間が3秒以内に完了し、ユーザー体験が損なわれない
- **SC-007**: 予測精度（R²値）と平均誤差が統計カードに表示されることで、運用担当者は予測の信頼性を一目で評価できる
- **SC-008**: 補充警告システムにより、在庫切れのリスクが50%以上削減される（運用データからの評価、長期指標）

## Assumptions *(前提条件)*

以下の前提条件に基づいて本仕様を策定する：

1. **データソース**: 予測データはCSVファイル（`/public/data/predictions.csv`）として提供され、定期的に更新される。リアルタイムAPI連携は現時点では不要。
2. **ユーザー権限**: すべてのユーザーは閲覧権限を持ち、認証・認可機能は不要（オープンダッシュボード）。将来的なアクセス制御の追加は想定される。
3. **データ品質**: CSVファイルは正しいフォーマット（列名一致、日付が昇順）で提供される。データクレンジングは軽微なエラーハンドリング（欠損値スキップ）のみ実施する。
4. **ホスティング環境**: GitHub PagesまたはVercelで静的サイトとしてホスティングされる。サーバーサイド処理は不要。
5. **ブラウザサポート**: モダンブラウザ（Chrome、Firefox、Safari、Edge の最新2バージョン）をサポート。Internet Explorer はサポート対象外。
6. **日付処理**: すべての日付は日本時間（JST、UTC+9）で処理・表示される。タイムゾーン変換機能は不要。
7. **データ更新頻度**: 予測データは日次または週次で更新される想定。リアルタイム更新や自動ポーリング機能は不要。
8. **補充閾値**: デフォルト値600 kgは運用実績に基づく標準値。ユーザーは自由に変更可能だが、値の妥当性検証は最小限（正の数値チェックのみ）。
9. **予測モデル**: 予測値はCSVファイルに事前計算されており、ダッシュボード側で再計算や予測モデルの実行は不要。
10. **表示期間**: 基準日を中心とした前後30日間（合計60日間）の表示が運用上最適であり、表示範囲の動的変更機能は不要。

---

## Out of Scope *(対象外機能)*

以下の機能は本仕様の対象外とする：

1. **ユーザー認証・認可**: ログイン、ユーザー管理、ロールベースアクセス制御は含まれない
2. **データ編集機能**: CSVデータの手動編集、補充履歴の追加・削除機能は含まれない
3. **予測モデルの再計算**: ダッシュボード側での予測アルゴリズムの実行、パラメータ調整は含まれない
4. **リアルタイムデータ連携**: API経由でのデータ取得、WebSocketによるリアルタイム更新は含まれない
5. **アラート通知**: メール、Slack、SMSなどの外部通知機能は含まれない
6. **データエクスポート**: グラフのPNG/PDF出力、CSVダウンロード機能は含まれない（将来的に追加可能）
7. **多言語対応**: 日本語以外の言語サポートは含まれない
8. **高度なデータ分析**: 統計的分析、相関分析、トレンド分析機能は含まれない
9. **カスタムダッシュボード**: ウィジェットの追加・削除、レイアウトのカスタマイズ機能は含まれない
10. **バックアップ・リストア機能**: データのバックアップ、履歴管理、バージョン管理は含まれない
