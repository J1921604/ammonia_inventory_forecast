## 2.1. 改行コード設定（Windows 環境での推奨設定）

### 🔧 改行コードの統一設定

**背景:**
- Windows（CRLF）と Unix/Linux（LF）では改行コードが異なります
- Git は "LF will be replaced by CRLF" という警告を出すことがあります
- リポジトリで改行ルールを統一しないと、無意味な差分やマージ競合が発生します

**推奨設定:**
1. リポジトリに `.gitattributes` を追加（チーム全体で統一）
2. ローカルで `core.autocrlf` を設定（Windows の推奨）

---

### 📝 .gitattributes の作成

**ルートディレクトリに `.gitattributes` ファイルを作成:**

```powershell
# .gitattributes を作成（UTF-8 で保存）
Set-Content -Path .gitattributes -Value @"
# 自動的にテキストファイルを正規化する
* text=auto

# Markdown や仕様書は LF を強制
*.md text eol=lf

# PowerShell スクリプトは CRLF を使用
*.ps1 text eol=crlf

# Shell スクリプトは LF を使用
*.sh text eol=lf

# JSON/YAML は LF を使用
*.json text eol=lf
*.yaml text eol=lf
*.yml text eol=lf

# バイナリファイルは変換しない
*.png binary
*.jpg binary
*.jpeg binary
*.gif binary
*.zip binary
*.exe binary
*.dll binary
"@ -Encoding UTF8
```

**目的:**
- テキストファイルの改行コードをリポジトリレベルで統一
- Markdown ファイル（仕様書等）は LF に統一
- PowerShell スクリプトは Windows 標準の CRLF を使用

---

### ⚙️ Git のローカル設定

```powershell
# Windows での推奨設定
git config --global core.autocrlf true
git config --global core.safecrlf warn

# 設定の確認
git config --global core.autocrlf
git config --global core.safecrlf
```

**設定の意味:**
- `core.autocrlf true`: チェックアウト時に CRLF へ変換、コミット時に LF へ変換
- `core.safecrlf warn`: 不可逆な変換が発生する場合に警告

---

### 🔄 既存ファイルの改行コードを正規化

```powershell
# 1. 現在の変更状態を確認
git status

# 2. インデックスをクリアして .gitattributes を反映
git rm --cached -r .

# 3. すべてのファイルを再登録（改行コードが正規化される）
git add .

# 4. 正規化をコミット
git commit -m "Normalize line endings with .gitattributes"
```

**注意:**
- `git rm --cached -r .` はインデックスから削除するのみで、作業ツリーのファイルは削除されません
- 大量の差分が表示される場合がありますが、これは改行コードの変更によるものです

---

### ✅ 確認方法

```powershell
# .gitattributes が作成されたことを確認
Test-Path .gitattributes

# Git の設定を確認
git config --list | Select-String "autocrlf|safecrlf"

# 次回以降、警告が出ないことを確認
git add .
# 警告が出なければ成功
```

**期待される結果:**
- `git add .` 実行時に "LF will be replaced by CRLF" 警告が出ない
- `.gitattributes` がリポジトリに含まれている

### 📝 完全な push 手順

```powershell
# 1. 現在のブランチ名を確認
git branch

# 2. ブランチ名を main に統一（必要な場合のみ）
git branch -M main

# 3. リモートリポジトリを追加
git remote add origin https://github.com/J1921604/ammonia_inventory_forecast.git

# 4. リモートの設定確認
git remote -v
# 出力例:
# origin  https://github.com/J1921604/ammonia_inventory_forecast.git (fetch)
# origin  https://github.com/J1921604/ammonia_inventory_forecast.git (push)

git push -u origin main
```

**目的:**
- ローカルの main ブランチをリモートの main ブランチに push
- `-u origin main` で今後の push/pull を簡略化

または:

#### ❌ エラー: "failed to push some refs"

**原因:** リモートに存在しない履歴がある（GitHub でファイルを追加してしまった場合）

**解決方法:**
```powershell

# コンフリクトが発生した場合は手動で解決後:
git add .
git commit -m "Merge remote changes"

# 再度 push
git push -u origin main
```

```powershell
# ブランチ名を main に変更

# 再度 push
git push -u origin main
```

### 1️⃣ `.specify/memory/constitution.md` - プロジェクト憲法

Spec Kit の初期化時に `.specify/memory/constitution.md` テンプレート自動生成。
このテンプレートを以下の内容で更新。

**憲法の内容（プロジェクト固有の原則）:**

```yaml
/speckit.constitution コード品質、テスト基準、ユーザー体験の一貫性、およびパフォーマンス要件に焦点を当てた原則を日本語で作成する

principles:
  - テスト駆動開発を徹底し、仕様に対する検証を必須とする
  - セキュリティ要件を機能要件より優先する
  - パフォーマンス閾値を定量化し、受入基準に組み込む

constraints:
  - 機密データの平文保存を禁止（暗号化・ハッシュ化必須）
  - 外部依存はバージョン固定により再現性を確保
  - 仕様と実装の乖離をレビューで検知・是正

governance:
  - 作業順序：憲法→仕様→計画→タスク→検証→実装→レビュー
  - 仕様と実装はブランチで分離（spec/impl）

ブランチ戦略:
  - 仕様策定用ブランチ
    git checkout -b feature/spec-<topic>
  - 実装用ブランチ（仕様ブランチから派生）
    git checkout -b feature/impl-<topic> feature/spec-<topic>

文字化け対策:
  - UTF-8エンコーディング

開発方針:
  - フロントエンドとバックエンドを同時に起動するコードも生成する
  - 繰り返し検証し、正常に動作するまでエラー修正を完了しない
```

**役割:** プロジェクト全体の原則・制約・ガバナンスを定義

#### ✅ ブランチ確認コマンド

```powershell
# 現在のブランチ名を確認
git branch
# または
git symbolic-ref --short HEAD

# すべてのブランチを確認（ローカル + リモート）
git branch -a

# ブランチの派生元を確認
git log --graph --oneline --all --decorate
```

**main ブランチへのマージ（オプション）:**

仕様ブランチ

```powershell
# main に切り替え
git checkout main

# 最新を取得
git fetch origin
git pull origin main

# 仕様ブランチをマージ
git merge 001-ammonia-forecast-dashboard

# リモートに反映
git push origin main

# ローカルブランチ削除
git branch -d 001-ammonia-forecast-dashboard

# リモートブランチ削除（必要なら）
git push origin --delete 001-ammonia-forecast-dashboard
```

⚠️ この場合、そのブランチでの作業内容は main に残りません。

```powershell
# マージせずに強制削除（履歴を残さない）
git branch -D 001-ammonia-forecast-dashboard
# リモートも消す場合
git push origin --delete 001-ammonia-forecast-dashboard
```

実装ブランチ

```powershell
# main に切り替え
git checkout main

# 最新を取得
git fetch origin
git pull origin main

# 実装ブランチをマージ
git merge feature/impl-001-ammonia-forecast-dashboard

# リモートに反映
git push origin main

# ローカルブランチ削除
git branch -d feature/impl-001-ammonia-forecast-dashboard

# リモートブランチ削除（必要なら）
git push origin --delete feature/impl-001-ammonia-forecast-dashboard
```

コンフリクトが起きた場合

```powershell
# 競合ファイルを確認
git status

# 手動で修正
git add
git commit

# 競合解決後
git push origin main
```

ローカル main の変更をリモートに反映（上書き）しない場合、慎重に`reset`（履歴が消えるため注意）。

```powershell
git fetch origin
git reset --hard origin/main
```
