# アンモニア在庫予測システム 憲法

## 核心原則

### I. テスト駆動開発の徹底（譲歩不可）

**原則**: すべての機能開発において、テスト駆動開発（TDD）を厳格に適用する。

**実装ルール**:
- テストコードを先に作成し、仕様に対する検証可能な受入基準を定義する
- ユーザー承認を得た後、テストが失敗することを確認してから実装を開始する
- Red-Green-Refactorサイクルを遵守する：テスト失敗 → 実装 → テスト成功 → リファクタリング
- 仕様書の受入シナリオごとに、対応するテストケースが存在しなければならない
- テストなしでのコミット、プルリクエストは受理しない

**根拠**: テスト駆動開発により、仕様と実装の乖離を防ぎ、リグレッションを最小化し、継続的な品質保証を実現する。

### II. セキュリティ優先の設計

**原則**: セキュリティ要件は機能要件よりも優先され、すべての設計判断において最優先事項とする。

**実装ルール**:
- 機密データの平文保存を絶対に禁止する（暗号化またはハッシュ化必須）
- 認証・認可機構は機能開発の基盤として最初に実装する
- すべてのユーザー入力に対してバリデーションとサニタイゼーションを実施する
- セキュリティイベントのログ記録を必須とする
- 脆弱性スキャンとセキュリティレビューを定期的に実施する

**根拠**: セキュリティ侵害は事業継続性を脅かし、ユーザー信頼を損なう。事後対応ではなく、設計段階からセキュリティを組み込むことで、リスクを最小化する。

### III. パフォーマンスの定量化と検証

**原則**: パフォーマンス要件は定量的な閾値として明示し、受入基準に組み込む。

**実装ルール**:
- すべての機能に対して、測定可能なパフォーマンス目標を設定する（例：応答時間 < 200ms、スループット > 1000 req/s）
- パフォーマンステストを継続的インテグレーションに組み込む
- ベースラインメトリクスを確立し、パフォーマンス劣化を検知する
- パフォーマンス閾値を超えた場合、本番環境へのデプロイを禁止する
- パフォーマンス要件をspec.mdに明記し、tasks.mdで検証タスクを定義する

**根拠**: ユーザー体験の中核はレスポンス速度であり、定量的な基準により一貫した品質を保証する。測定なくして改善なし。

## 制約事項

### セキュリティ制約

- 機密データ（パスワード、API キー、個人情報）の平文保存を禁止
- データベースに格納するパスワードは bcrypt、Argon2 等の強力なハッシュアルゴリズムで保護
- API キー、シークレットは環境変数または専用のシークレット管理サービスで管理
- 通信は HTTPS/TLS で暗号化（開発環境を除く）

### 依存関係管理

- すべての外部依存ライブラリはバージョンを固定（package.json、requirements.txt 等）
- 範囲指定（^、~）を避け、完全な再現性を確保
- 依存ライブラリの更新は意図的な変更としてバージョン管理に記録
- セキュリティパッチ適用時は、テストスイートで影響範囲を検証

### 仕様・実装の整合性

- 仕様書（spec.md）と実装コードの乖離は、コードレビューで検知・是正する
- 実装中に仕様変更が必要になった場合、spec.md を先に更新してから実装を変更
- すべてのプルリクエストは、仕様書との整合性チェックを必須とする
- レビュアーは、実装が仕様の受入基準を満たしているか確認する責任を持つ

### エンコーディング標準

- すべてのソースコード、設定ファイル、ドキュメントは UTF-8 エンコーディングで保存
- BOM（Byte Order Mark）なしの UTF-8 を使用
- Git の core.autocrlf を適切に設定し、改行コードの不整合を防止

## ガバナンス

### 開発ワークフロー（作業順序）

すべての機能開発は、以下の順序を厳守する：

1. **憲法確認**: `.specify/memory/constitution.md` で原則・制約を確認
2. **仕様策定**: `specs/[###-feature]/spec.md` でユーザーシナリオ・要件を定義
3. **計画作成**: `specs/[###-feature]/plan.md` で技術アプローチ・構造を設計
4. **タスク分解**: `specs/[###-feature]/tasks.md` で実装タスクを列挙
5. **テスト作成**: 受入基準に基づきテストコードを先行実装
6. **検証（Red）**: テストが失敗することを確認
7. **実装（Green）**: テストを通過させる最小限の実装
8. **リファクタリング**: コード品質を向上させる
9. **レビュー**: 仕様整合性・憲法準拠を確認してマージ

### ブランチ戦略

**仕様策定フェーズ**:
```
git checkout -b feature/spec-<topic>
```
- 仕様書、計画書、タスク一覧を作成
- レビュー承認後、main にマージ

**実装フェーズ**:
```
git checkout -b feature/impl-<topic> feature/spec-<topic>
```
- 仕様ブランチから派生し、実装とテストを追加
- 憲法・仕様への準拠を確認後、main にマージ

**ブランチ命名規則**:
- `feature/spec-<topic>`: 仕様策定用
- `feature/impl-<topic>`: 実装用
- `bugfix/<issue-number>-<description>`: バグ修正用
- `hotfix/<critical-issue>`: 緊急対応用

### レビュープロセス

**必須レビュー項目**:
1. 憲法準拠チェック（3つの核心原則への違反がないか）
2. 仕様整合性チェック（spec.md の受入基準を満たしているか）
3. テストカバレッジ確認（すべての受入シナリオにテストが存在するか）
4. セキュリティレビュー（制約事項への違反がないか）
5. パフォーマンス検証（定量的な閾値を満たしているか）

**承認基準**:
- すべての CI テストがパス
- 少なくとも 1 名のレビュアーによる承認
- 重大変更（破壊的変更、セキュリティ影響大）は 2 名以上の承認を必須とする

### 違反対応

憲法違反が発見された場合：
1. プルリクエストを即座にブロック
2. 違反内容と影響範囲を文書化
3. 是正計画を作成し、レビュアーの承認を得る
4. 是正後、再度レビュープロセスを実施

## 開発環境設定

### 同時起動設定

フロントエンドとバックエンド（存在する場合）を同時に起動するスクリプトを用意する：

```json
{
  "scripts": {
    "dev": "concurrently \"npm run dev:frontend\" \"npm run dev:backend\"",
    "dev:frontend": "next dev",
    "dev:backend": "node backend/server.js"
  }
}
```

現行プロジェクトは Next.js のみのため、`npm run dev` で起動。

### 検証サイクル

- 開発中は繰り返し検証を実行し、エラーが発生した場合は正常動作まで修正を完了する
- エラーログを完全に解消するまで次のタスクに進まない
- デプロイ前に、本番環境相当での統合テストを必須とする

## 憲法改訂

本憲法の改訂は以下の手順で実施する：

1. 改訂提案を `.specify/memory/constitution.md` に記録
2. 影響を受けるテンプレート・ドキュメントを特定
3. チーム全体でのレビューと合意形成
4. バージョン番号を更新（セマンティックバージョニング）
   - MAJOR: 後方互換性のない原則の削除・再定義
   - MINOR: 新しい原則・セクションの追加
   - PATCH: 文言の明確化、タイポ修正
5. 関連ドキュメントを一括更新
6. 改訂履歴を記録
